# BZZ Decoder Plan

`djvulpes` should stay MIT-licensed. The in-house BZZ decoder must therefore be developed without copying or mechanically porting GPL-covered DjVuLibre code or ExifTool's BZZ implementation.

## Constraints

- Do not copy DjVuLibre or ExifTool BZZ source code.
- Do not copy large static probability/adaptation tables from GPL-covered implementations.
- Use compressed/decompressed fixture pairs as behavioral tests.
- Use the local `bzz` command only as a temporary test oracle and fallback.
- If the decoder needs generated constants, keep the generator in this repo and document the source formula or derivation.
- The current bit-model table generator is a provisional in-repo scaffold. It must be replaced or validated against an independently derived BZZ-compatible model before removing the external `bzz` fallback.
- The provisional split table is generated from a linear `state / states` baseline plus a small concave boost. The boost was chosen from fixture constraints to move the first known entropy mismatch forward; it is not copied from DjVuLibre, ExifTool, or any static table.
- The provisional state transition step is currently `3` states per observed bit. Fixture diagnostics showed that the earlier step of `4` adapted too quickly after repeated zero observations.
- Block decoding now consumes the two pass-through `FSHIFT` bits after the block length and before the MTF-number stream.
- MTF-number entropy decoding now uses the spec-shaped 262-context layout with previous-`MTFNO` selectors and tree-shaped `decode_bin` context paths. The Z′ state model and BZZ frequency-augmented MTF update are still provisional/incomplete.
- A test-gated, spec-literal Z′ register-machine scaffold now verifies pass-through decoding against the fixture header and exercises the MPS/LPS update recurrence with scripted table values. The fixed Z′ tables still need to be sourced or derived independently before this can replace the provisional bit model.
- `djvu-zp` is used as a dev-only MIT clean-room oracle to validate the Z′ scaffold and public table semantics. It is not a runtime dependency. The oracle tests compare decoded bits and context transitions across multiple byte patterns and context reuse patterns without copying its tables into runtime code.
- Test code now routes Z′ table access through an owned `ZpTableSet` generated by a `ZpTableGenerator`. For now the candidate generator is dev-oracle backed; the next replacement point is an independent generator that produces the same shape and passes the same oracle comparison tests.
- Candidate table tests now capture spec-level structure separately from exact oracle equality: early/steady threshold phases, the first fresh-context jump into late estimation, and paired early-estimation probability/threshold entries.
- `theta` now has a dedicated candidate hook plus an ignored residual diagnostic. A simple asymptotic estimate matches the small-delta tail closely but is still far from the oracle in the first early-estimation states, so the next derivation work should focus on that high-delta region.
- Rank-to-symbol reconstruction now uses BZZ's frequency-augmented MTF rotation with per-block `FSHIFT`, rather than plain move-to-front. A true-Z′ rank-stream diagnostic still mismatches the fixture at the first rank (`5` decoded vs `10` expected), so the next functional blocker is rank bit layout or stream position, not plain-MTF reconstruction.
- The MIT `djvu-zp` source confirms that BZZ pass-through uses `0x8000 + A/2`; `0x8000 + 3A/8` is the separate IW44 pass-through mode. A mixed pass-through-plus-context oracle test now verifies our literal Z′ stream position after the 24 block-size bits and two `FSHIFT` bits. The first true-Z′ rank path diverges from the BWT-derived expected path at selector context `8`.
- Additional diagnostics show nearby raw-bit alignment variants and simple selector-layout variants can make the first rank match, but none match beyond the first rank. These are not sufficient fixes; keep them as diagnostics while investigating the remaining rank-stream mismatch.

## Implementation Stages

1. Decode BZZ block framing with an in-memory bit reader.
2. Implement the binary entropy model from independently derived constants.
3. Decode the rank/MTF symbol stream for one block.
4. Implement inverse Burrows-Wheeler reconstruction.
5. Decode all blocks and remove the external `bzz` fallback.

The current fixture target is `tests/fixtures/bzz/hello.bzz`, which should decode exactly to `tests/fixtures/bzz/hello.raw`.
